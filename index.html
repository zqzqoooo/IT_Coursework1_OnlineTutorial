Page 4: Intro to Vue.js (Carving vs. LEGOs)

Okay, let's officially start by introducing what Vue.js is.

To help you understand Vue most intuitively, I want you to imagine two scenarios: carving wood and building with LEGOs.

Native web development, which we're all familiar with, is like 'carving wood'. You start with a raw block of wood (empty HTML and JS), and you must act as a craftsman, holding a chisel (commands like getElementById, createElement), and meticulously carve the page, step-by-step, into what you want. This process is precise but also very tedious. You have to care about every single step. This is what we call 'imperative' development.

Vue.js, on the other hand, completely changes the game. It's like 'building with LEGOs'. Vue has already prepared a huge box of standardized, functional 'bricks', which in the Vue world, we call 'Components'.

Your job is no longer to carve, but to 'assemble'. You just need to 'declare': 'I need a header (Brick A), a sidebar (Brick B), and a comment section (Brick C)', and then you snap them together. You care about 'what I want'—the final result—not the tedious process of 'how to get there'. This is 'declarative' development.

This clever 'brick system' was created by ex-Google engineer Evan You. Its biggest feature is that it's 'Progressive', meaning you can use these 'LEGOs' to build a giant castle (a complex Single Page App), or you can just take one tiny brick to decorate your original 'wood carving' (add a bit of interactivity to an old project).

Page 5: Declarative Development

We just mentioned 'declarative', and this concept is the soul of Vue, so let's dive deeper.

'Declarative' development fundamentally changes your focus. In the past, if a piece of data changed, you had to worry about two things: 1. Updating the data, and 2. Manually writing code to update the page.

Look at this diagram. In the world of Vue, the only thing you need to care about is the 'Data' (your JavaScript object). For example, you define a piece of data called count with a value of 0.

Then, you only need to 'declare' once in your HTML: 'Hey, the content of this <h1> tag, please stay in sync with my count data forever.' This process is called 'Binding'.

From this moment on, something magical happens: you never have to touch that <h1> tag again. Your entire job is now just to modify the count data.

When you execute count++, Vue's 'Reactivity System' automatically detects: 'Oh! count changed!' It then immediately and automatically updates the <h1> tag to show 1.

This is 'Data-Driven View'. Your job shifts from tedious 'wood carving' (manual DOM manipulation) to clear 'data management' (managing your JS objects). This massively reduces our code, makes the logic clearer, and is far less error-prone.

Page 6: Progressive Framework

What does 'progressive' mean? It means Vue is incredibly flexible. Like our LEGOs, you can decide how much of it to use based on your needs.

First way: The simplest. Imagine you already have a website. You don't want to rewrite it, you just want one part, like a comment section, to be smarter. No problem. You add one <script> tag to your HTML to include Vue, and you can immediately enjoy 'declarative' benefits. This is perfect for enhancing old projects, and it's what we'll do in our tutorial.

Second way: Your goal is to build a complete 'Single Page App' (SPA), like GMail, where clicking links doesn't refresh the whole page. Vue can do this too. It provides official plugins like Vue Router to manage page navigation.

Third way: Your goal is to build a massive, complex enterprise app like GitLab. Vue can also do this. It offers a full 'ecosystem' of tools, like Pinia (for managing shared data) and Vite (for lightning-fast development and building).

So, the core advantage of 'progressive' is: it's very easy to start with, but its ceiling is extremely high. This flexibility is something many other frameworks lack.

Page 7

Page 8: Why use Vue? (vs. Native)

So, to summarize our previous two core ideas, 'Why do we choose Vue?'

The fundamental reason is that it brings a complete shift in mindset.

Native JavaScript is 'imperative', like 'carving wood'. You must care about every step of 'how to do it': manually find the DOM element, manually update it. Data and view are separate, and you write a lot of code to 'command' them to stay in sync.

Vue.js is 'declarative', like 'building with LEGOs'. You only care about 'what to do'—your desired result. You 'bind' the data and the view together, and then you just modify the data. Vue's reactivity system automatically handles all the tedious DOM operations for you.

This shift frees us from the tedious 'process' and lets us focus on what's truly important: the 'data' and the 'state'. This is why Vue dramatically improves development efficiency and code maintainability.

Page 9: Vue's Industry Position (Usage & Community)

A framework can be great, but if no one uses it, it's just a toy. So what is Vue's status in the real world?

The data is astounding. According to the authoritative "State of JavaScript 2023" survey, Vue.js is the second most-used front-end framework globally, with over half of all respondents using it.

More importantly, Vue has an extremely strong and healthy 'ecosystem'. When you build large apps, Vue provides the best-in-class, official solutions for routing (Vue Router) and state management (Pinia).

Its community is incredibly active, and there are more learning resources than you'll ever need.

Finally, look at who uses it. From Alibaba and GitLab to Nintendo, and even parts of projects at Netflix and Facebook. The adoption by these top-tier companies fully validates Vue's stability and reliability in large-scale projects.

Page 11: What Do You Need to Learn Vue?

After talking about so many benefits of Vue, you might be thinking: 'This framework is so powerful, is it hard to learn?'

This is the best news: Vue is famous for its gentle learning curve. The prerequisites to start learning Vue are exactly what we've been learning in this course: the 'web trio'.

  1. HTML: You need to be familiar with HTML. Vue's templates are written in HTML; it's like "super HTML".
  2. CSS: You need to understand CSS, because the final page still needs CSS for styling.
  3. JavaScript: Most importantly, you need to grasp the fundamentals of JavaScript. Things like variables, functions, objects, and array manipulation. You don't need to be a JS master to start, but the more solid your foundation, the faster you'll learn.

So, the knowledge you are gaining right now is the perfect foundation for becoming a Vue developer.

Page 12: Comparison (1) - "Modular" Page Structure

Let's look at a real example: a page with a header, sidebar, and comment section.

Native HTML (left placeholder): Very direct, but also very 'flat' and 'bloated'. All the HTML for the header, sidebar, and comments is piled into one file. As the page gets more complex, this file becomes hundreds of lines long.

Vue (right placeholder): Now look at the <body> organized with Vue. It's incredibly clean. It's like a 'table of contents'. You can instantly understand: 'Okay, this app is made of a <TheHeader>, a <TheSidebar>, and a <CommentSection>.'

Vue lets us package an entire block of HTML into a custom tag, like <TheHeader>. This is the visual representation of 'structural modularity'.

Page 13: Comparison (2) - "Modular" Logic

This 'modularity' isn't just in the HTML; it's even more critical in the JavaScript logic.

Native JS (left placeholder): All our logic—header logic, comment logic—is mixed together in one giant file. They share global variables and get tangled. This is 'high coupling'.

Vue (right placeholder): Every 'LEGO brick' (component) is an independent 'box'. The TheHeader component is its own little world. The CommentSection component is also its own world, managing its own comment list (data) and add-comment function (methods).

They don't interfere with each other. This is 'high cohesion, low coupling' and the essence of our 'LEGO' metaphor.

This modular thinking makes our code extremely easy to maintain, test, and reuse. If you want a comment section on another page, you just drop in the <CommentSection> brick.

  • It moves us from 'imperative' to 'declarative', freeing us from 'carving wood' (tedious DOM operations) to focus on what matters: 'data' and 'state'.
  • It lets us 'build with LEGOs' using a powerful 'Component System' to create complex, maintainable applications in a modular way.
  • Finally, it has a strong, friendly community and is built on the HTML, CSS, and JS fundamentals you are already learning in this course.

Part 3: Introduction to Core Modules (Composition API)

Before building, you must understand Vue's core concepts.

1. Single File Component (SFC)

A Vue project is made of .vue files. Each file is a component, containing three parts:

  • <template>: Holds the HTML structure.
  • <script setup>: Holds the JavaScript logic (All our code will go here).
  • <style scoped>: Holds the CSS styles (scoped means the styles only apply to this component).

2. Reactivity (ref)

This is Vue's "magic".

  • Function: ref converts a normal JS variable (like a string, number, or array) into a "reactive reference".
  • Usage: When you change the value of this reference, Vue automatically updates every part of the <template> that uses it.
  • Note: In <script>, you must use .value to access or modify its value.
  • Example:
import { ref } from 'vue'
const message = ref('Hello')
console.log(message.value) // "Hello"
message.value = 'Goodbye'  // This will trigger an update

3. Template Syntax

  • Interpolation ({{ }}): <h1>{{ message }}</h1>. Used to display a ref's value in the HTML.
  • Attribute Binding (:): Short for v-bind. Used to dynamically bind HTML attributes. 
    <!-- Wrong: 'my-class' is a plain string -->
<div class="my-class"></div>
<!-- Correct: :class is bound to the JS variable myClassName -->
<div :class="myClassName"></div>
  • Event Listening (@): Short for v-on. Used to listen for DOM events (like a click). 
    <button @click="myFunction">Click Me</button>

4. Common Directives

  • v-model (Two-Way Binding): The perfect partner for ref. It "binds" a form input to a ref variable in both directions. 
    <input v-model="message">

    (Now, the input's content and message.value will always be in sync.)

  • v-if / v-else (Conditional Rendering): 
    <p v-if="events.length > 0">You still have events.</p>
<p v-else>No events left.</p>
  • v-for (List Rendering): 
    <ul>
  <li v-for="event in events" :key="event.id">
    {{ event.name }}
  </li>
</ul>

    ( key is required. It helps Vue efficiently track each item.)

5. Component Communication

  • defineProps: How a child component receives data from its parent.
  • defineEmits: How a child component sends a message (emits an event) to its parent (e.g., "Please delete me!").

6. Lifecycle (onMounted)

onMounted is a "lifecycle hook" that runs automatically after the component has been rendered to the page. It's the perfect time to fetch initial data from a server or local storage.

7. Watcher (watch)

watch is used to "watch" a ref. As soon as that ref's value changes, the watch function will run. It's the perfect time to save data.

Step 1: App.vue (The Parent Component)

Open src/App.vue and replace all of its content with the following code. This is your "main" component.

<!-- src/App.vue -->
<script setup>
// Modules: Import all needed functions
import { ref, onMounted, watch } from 'vue'
// Modules: Import the child component
import EventItem from './components/EventItem.vue'

// --- State Definition ---
const newEventName = ref('')
const events = ref([]) // Start with an empty array
const STORAGE_KEY = 'vue-highland-events'

// --- Data Persistence ---

// Modules: Lifecycle (onMounted)
// 1. This function runs automatically when the component loads
onMounted(() => {
  const savedEvents = localStorage.getItem(STORAGE_KEY)
  if (savedEvents) {
    events.value = JSON.parse(savedEvents)
  }
})

// Modules: Watcher (watch)
// 2. Watch the 'events' array. As soon as it changes, save it.
watch(events, (newEventList) => {
  localStorage.setItem(STORAGE_KEY, JSON.stringify(newEventList))
}, { 
  // 'deep: true' is essential for watching inside the array
  deep: true 
})

// --- Core Logic (Functions) ---

// 3. Function to add a new event
function addEvent() {
  if (newEventName.value.trim() === '') return
  
  events.value.push({
    id: Date.now(), // Use a timestamp as a unique ID
    name: newEventName.value
  })
  
  newEventName.value = '' // Clear the input box
}

// 4. Function to handle the delete request from a child
function handleDelete(eventId) {
  events.value = events.value.filter(event => event.id !== eventId)
}
</script>

<template>
  <main>
    <h1>Highland Games Event Planner</h1>
    
    <form @submit.prevent="addEvent">
      <input 
        v-model="newEventName" 
        placeholder="Enter a new event...">
      <button type="submit">Add Event</button>
    </form>
    
    <h2>Event List</h2>
    
    <ul v-if="events.length > 0">
      <EventItem 
        v-for="event in events" 
        :key="event.id"
        :event="event"
        @delete-event="handleDelete"
      />
    </ul>
    
    <p v-else>
      All clear! Add your first event.
    </p>

  </main>
</template>

// Modules: Props and Emits

const props = defineProps({

Part 5: Final Code Dissection

You've built the app. Now, let's understand exactly why it works.

1. App.vue (The Parent Component / State Manager)

App.vue acts as the "brain" of your project. It manages all the data (state) and core logic.

App.vue <script setup> Dissection:

  • import { ref, onMounted, watch }
    • ref: Creates all reactive data (newEventName, events).
    • onMounted: Loads Data. A lifecycle hook that runs once when the component first loads. We use it to read from localStorage, enabling persistence.
    • watch: Saves Data. A listener that "watches" the events array. As soon as events changes (add or delete), the watch function runs, saving the new array to localStorage. { deep: true } is essential for it to watch inside the array.
  • const events = ref([])
    • This is the "Single Source of Truth" for the application. All events are stored here.
  • function addEvent() { ... }
    • This function mutates the events array by .push()ing a new object with a unique ID (Date.now()).
  • function handleDelete(eventId) { ... }
    • This function also mutates the events array. It uses .filter()—a modern, immutable JS practice. It doesn't modify the original array but returns a new array without the deleted item. Vue detects that events.value was assigned a new array and updates the view.

App.vue <template> Dissection:

  • <form @submit.prevent="addEvent">
    • @submit.prevent: This is beautiful Vue syntax. @submit listens for the form submission (like pressing Enter). .prevent is a modifier that automatically calls event.preventDefault(), stopping the page from reloading.
  • <EventItem ... /> (The most critical interaction)
    • v-for="event in events" :key="event.id": Loops through the events array, creating one EventItem component instance for each object. :key is required for performance.
    • :event="event": "Props Down". The parent App.vue passes the individual event object down to the child EventItem via a prop named event.
    • @delete-event="handleDelete": "Events Up". The parent App.vue listens for a custom event named delete-event emitted by the child. When it "hears" this event, it calls its own handleDelete function.

2. Example of Angular Code:

EventItem.vue is a "dumb" component (or "presentational component"). It doesn't manage any state. Its only jobs are:

  1. Display the data it's given (via props).
  2. Tell the parent when something needs to happen (via emit).

EventItem.vue <script setup> Dissection:

  • const props = defineProps({ event: Object })
    • !! Critical Fix !! defineProps is a macro that returns an object containing all declared props. We must assign this to a props constant. This is how we access the data inside our <script> logic (e.g., in the requestDelete function) using props.event.
  • const emit = defineEmits(['delete-event'])
    • This line declares all events this component is allowed to emit. It tells Vue, "I might emit a 'delete-event' at some point."
  • function requestDelete() { ... }
    • A local function called when the user clicks the "Delete" button.
    • emit('delete-event', props.event.id): This is the "Event Up" in action. It emits the event we declared, and attaches a payload (props.event.id). This payload is passed directly to the parent's listener function (i.e., App.vue's handleDelete(eventId)).

Data Flow Summary:

  1. Data Flow: App.vue → events array → v-for → :event prop → EventItem.vue (Data flows down).
  2. Event Flow: EventItem.vue (User Click) → requestDelete() → emit('delete-event', id) → App.vue (@delete-event="handleDelete") → handleDelete(id) function (Events flow up).
